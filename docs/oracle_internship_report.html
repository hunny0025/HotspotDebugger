Executive Summary

ORACLE (Over-the-air Reconnaissance and Acquisition for Connected Log Environments) is a production-grade, forensically sound Android acquisition and analysis platform built entirely in Rust. Developed as part of the GPCSSI 2024 internship program at Gurugram Cyber Police, this platform represents a fundamental advancement in automated mobile device forensics.

ORACLE acquires wireless and connectivity evidence from live Android devices or offline filesystem images, applies a multi-layer mathematical reasoning engine, and produces court-ready PDF reports with embedded mathematical justifications for every forensic finding. The platform addresses critical gaps in existing mobile forensic toolchains — particularly around BFU (Before First Unlock) state handling, cryptographic chain-of-custody, and deterministic evidence scoring.

✅
Core Guarantee: Given identical inputs, ORACLE will always produce bit-identical output. No randomness, no hidden state, no external drift. Every byte is accounted for, every decision is logged.

Forensic Pipeline Walkthrough

The ForensicPipeline orchestrator in oracle-cli executes 8 sequential phases. Each phase is atomic — a failure at any step causes a hard halt with a forensically descriptive error, never a silent continuation.

1

🔌 Pre-Flight Verification

ADB binary version recorded in audit log. Exactly one device matching the target serial must be present. Device must respond to echo test (authorization verified). Failure → HARD HALT — investigation cannot proceed.

adb.verify_connection()serial matchingecho test
2

🔒 Capability Detection (3-Tier BFU Probing)

Scans ro.crypto.state + ro.crypto.type, then vold.decrypt, then live CE path probe on /data/user/0/. If BFU detected, examiner is prompted with 3 choices before any extraction.

DeviceEncryptedCredentialEncryptedDeAndCeExaminerDecision
3

📦 Artifact Acquisition & Integrity Sealing

ADB pull → raw bytes → SHA-256 computed → hash written to audit log entry → THEN bytes passed to parser. This is the write-before-execute guarantee. No re-computation, ever.

SHA-256AuditLogWriterContentAddressableStore
4

🔍 Multi-Parser Extraction

5 specialized parsers run against the sealed artifacts. Each returns structured ParsedOutput records with byte offsets, baseline confidence, and inline anomaly flags.

WifiConfigStoreParserWpaSupplicantParserDhcpLeaseParserConnectivityLogParserSamsungWifiLogParser
5

⚙️ Normalization & Conflict Detection

All records unified into a standard schema. Normalizer detects: SsidMismatch, BssidMismatch, SecurityProtocolMismatch. Conflicts flag both records as mutually contradicted.

6

📅 Timeline Correlation

Sessions assembled from events. Timeline engine detects: simultaneous connections (overlap > 300s = WARNING), extended gaps (>24h), and temporal order violations (events in impossible chronological sequence).

7

📊 Confidence Scoring

Confidence Model v2.0.0 applied. Every finding receives a final score between 0.0 and 1.0, a classification (Definitive/High/Moderate/Low/Insufficient/Contradicted), and a full reasoning chain for court use.

8

📋 Report Generation

Deterministic PDF generated using printpdf. Report receives a SHA-256 integrity seal on the final page. JSON evidence export also generated for programmatic consumption.

Confidence Scoring Engine — Full Technical Detail

The confidence engine is the mathematical heart of ORACLE. It converts raw parsed records into court-defensible confidence assessments. The engine implements Confidence Model v2.0.0.

5.1 The Core Formula
C_final = clamp(0.0, 1.0,
    ((W_SR × S_R + W_CS × C_S) × T_T × A_V × C_V × (1 − A_F)) − C_P

5.3 Corroboration Score (C_S) — Logarithmic Curve
fn compute_corroboration(source_count: usize) -> f64 { if source_count <= 1 { 0.0 // No corroboration — single-source finding } else { let n = source_count as f64; ((n - 1.0) / 3.0).min(1.0) // Logarithmic cap — prevents redundancy inflation } }
Sources CorroboratingC_S ValueCourt Meaning

2 sources0.33Single corroborating source
3 sources0.67Two corroborating sources — strong pattern
4+ sources1.00Full corroboration (capped — prevents over-inflation)

5.5 Worked Example — Scoring a Wi-Fi Connection
📡
Scenario: Device connected to "CoffeeShop_5G" found in WifiConfigStore, DHCP log, and Samsung log.
S_R = 0.95 (WifiConfigStore — highest reliability class) C_S = 0.67 (3 sources: WifiConfigStore + DHCP + Samsung OEM) Raw base = (0.60 × 0.95) + (0.40 × 0.67) = 0.57 + 0.268 = 0.838 Timestamp: UnverifiedLocal → T_T = 0.80 Volatility: Persistent → A_V = 1.00 Anti-forensics: None detected → A_F = 0.0 Contradiction penalty: None → C_P = 0.0 Scaled = 0.838 × 0.80 × 1.00 × 1.00 × (1.0 - 0.0) = 0.670 C_final = clamp(0, 1, 0.670 - 0.0) = 0.670 Classification: MODERATE Reasoning Chain (embedded in PDF): • Primary evidence from WifiConfigStore (reliability: 0.95) • Corroborated by 3 sources — DHCP + Samsung OEM (C_S: 0.67) • Timestamp trust: UNVERIFIED_LOCAL (T_T: 0.80) • No anti-forensic indicators detected • Final score: 0.670 — Classification: MODERATE

6.1 Temporal Order Violation Detection (Rust Code)
fn check_temporal_order(sessions: &[TimelineSession], anomalies: &mut Vec<Anomaly>) { for session in sessions { for window in session.events.windows(2) { if window[0].timestamp > window[1].timestamp { anomalies.push(Anomaly { category: AnomalyCategory::TemporalOrderViolation, severity: AnomalySeverity::Critical, description: format!( "Event at {} occurs AFTER subsequent event at {} — violates temporal causality.", window[0].timestamp, window[1].timestamp ), recommendation: "Verify acquisition timestamps and check for clock manipulation on the device.", }); } } } }

Parser Layer — Forensic Validation
7.1 ZeroRecordReason — Never Silent Failure

If a valid artifact yields zero parsed records, ORACLE does NOT silently ignore it. It categorizes the reason. A PossibleWipe is immediately escalated as a CRITICAL anti-forensic indicator.

pub enum ZeroRecordReason { EmptyArtifact, // Zero-length file — device wiped? ValidButEmpty, // Valid format, legitimately no records Truncated, // File ends prematurely — incomplete write PossibleWipe, // File present but content zeroed — ANTI-FORENSIC INDICATOR UnrecognizedFormat, // Schema mismatch — wrong Android version? AllRecordsInvalid, // All lines failed validation }
7.2 AnomalyFlag — Per-Record Inline Flags
pub enum AnomalyFlag { FutureTimestamp, // Timestamp is AFTER acquisition time AncientTimestamp, // Timestamp is year 1970 or similar RandomizedMac, // MAC locally-administered bit set (device randomization) DuplicateRecord, // Same record appears at two byte offsets OrphanNetwork, // Network not seen in any other artifact NonChronologicalOrder, // Timestamp breaks chronological sequence SuspiciousSsid, // SSID contains non-printable characters UnexpectedIpRange, // Link-local IP where DHCP range expected AbnormalLeaseDuration, // Lease impossibly short (<60s) or long (>30 days) Custom(String), // Freeform examiner annotation }
7.3 ParsedOutput — Universal Parser Output Structure
pub struct ParsedOutput { pub record_type: String, // e.g., "dhcp_lease" pub record_data: serde_json::Value, // The extracted payload (schema-versioned) pub byte_offset: Option<u64>, // Where in the file this record lives pub byte_length: Option<u64>, // How many bytes this record spans pub confidence: f64, // Parser's own baseline confidence pub anomaly_flags: Vec<AnomalyFlag>, // Flags raised during parsing }
⚖️
The byte_offset and byte_length fields allow an expert witness to point to the exact location in the raw evidence file where a finding originated — critical for court testimony.

Report Generation — What Gets Output

The oracle-report crate uses printpdf (no external font files, no runtime dependencies) to generate a fully deterministic PDF with the following structure:

📄 Page 1: Title Page
Case number, Report ID, Investigation ID, Examiner Name, Organization, Badge ID, Generation timestamp, CONFIDENTIAL marking.
📊 Page 2: Executive Summary
Total artifacts acquired, findings count, high-confidence count, contradicted count, anomalies detected.
🔍 Page 3+: Findings
Finding number, title, confidence classification, SSID/BSSID, protocol, timestamp, corroboration count, and full reasoning chain.
📋 Evidence Appendix
SHA-256 hash of every raw artifact, original device path, acquisition timestamp, file size in bytes.
🔬 Methodology Disclosure
Confidence Model v2.0.0 documentation, factor weights, and legal statement for court admittance.
🔏 Integrity Seal
SHA-256 hash of the full report content on the last page, allowing verification that the report was not altered after generation.

Conclusion & Impact

Project ORACLE represents a significant advancement in automated Android mobile forensics. By combining strict cryptographic integrity, advanced BFU state awareness, mathematical confidence scoring, intelligent anomaly detection, and transparent reasoning chains into a memory-safe compiled binary, it provides DFIR professionals and law enforcement agencies with a defensible, highly reliable toolset for investigating connected log environments.

⚖️
Court-Defensible Output
Every confidence score is mathematically justified. Examiners can testify to the exact formula applied, the weights used, and the sources corroborated.
🚀
Paradigm Shift in Mobile Forensics
ORACLE eliminates the "black box" problem in forensic tools — every decision is logged, every byte is hashed, every conclusion is explained.
🔐
Zero Trust Architecture
No external network dependencies. No runtime. No black-box libraries. Deployable in the most secure, air-gapped forensic lab environments.
📈
Extensible Design
New OEM parsers, new anomaly checks, and new artifact classes can be added as Rust modules without touching the core scoring engine.
